# 实现计划：逐岗位即时筛选与沟通

## 概述

将现有 `scan -> REVIEW batch -> apply` 两阶段主流程改为逐岗位流式流程：在岗位详情页完成信息读取、硬过滤、简历匹配和立即沟通；成功沟通每满 5 个冷却 120 秒，并按北京时间持久化当日成功数，达到 150 后停止。

## 架构决策

- 当前岗位列表代表目标城市，不在详情页重复校验地址。
- 薪资按月薪区间下限判断，最低 15K；无法解析或面议时保守跳过。
- HR 仅允许今日或近 3 日活跃；硬件、嵌入式、物联网和车载等方向在评分前否决。
- Appium 适配器暴露“读取下一个详情岗位”和“沟通当前岗位”，避免返回列表后重新查找。
- 只有新沟通成功才写入持久化台账并增加当日计数；已沟通、失败和跳过不计数。
- 遇到安全验证、登录异常或平台限制立即停止，不自动绕过。

## 风险和缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 点击沟通后进入聊天页，列表状态变化 | 高 | 下一次读取前统一恢复岗位列表，同时在适配器内保留已见指纹 |
| 薪资格式多样 | 中 | 只支持可证明的 K/万元区间，未知格式保守跳过 |
| 岗位领域关键词误杀 | 中 | 强特征单词与组合规则分开；“语音机器人平台”和移动 App 测试不因弱词排除 |
| 重启导致计数丢失 | 高 | 每次成功后立即追加持久化台账 |
| 平台安全验证或沟通限制 | 高 | 抛出人工处理错误并停止，不自动重试绕过 |
